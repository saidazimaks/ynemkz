# Выгодный Город — Архитектура системы

Telegram-бот + Mini App: локальный дисконт-клуб (Экибастуз). Роли: покупатель, партнёр, админ.

**Стек:** Python 3.12, aiogram 3.x, Telegram Mini App (React/vanilla), Supabase (Postgres + Auth + Storage), деплой — VPS или Railway/Fly.io.

---

## 1. Компоненты

```
Telegram ──► aiogram-бот (Python) ──► Supabase Postgres
   │              │                        ▲
   │              ├─ QR-генератор          │
   │              └─ Планировщик           │
   │                 (APScheduler)         │
   └─► Mini App (WebApp) ── REST/Supabase ─┘
              (каталог скидок, кабинет партнёра)
```

| Компонент | Ответственность |
|---|---|
| **aiogram-бот** | Регистрация, платежи (чек), выдача QR, ввод кода партнёром, админ-команды, рассылки |
| **Mini App** | Витрина скидок (красивый каталог), кабинет партнёра со статистикой, админ-панель |
| **Supabase Postgres** | Все данные. RLS не нужен на старте — бот ходит через service key |
| **Supabase Storage** | Скрины чеков Kaspi, логотипы партнёров |
| **APScheduler** | Смена скидки дня, деактивация просроченных подписок, розыгрыш по расписанию, протухание QR |

Что в боте, а что в Mini App: **все действия** (оплата, QR, ввод кода) — в боте, кнопками. **Просмотр** (каталог скидок, статистика, списки) — в Mini App. Так MVP работает даже без Mini App вообще.

## 2. Схема БД

```sql
users (
  id            bigint PK,            -- telegram_id
  username      text,
  full_name     text,
  phone         text,
  role          text DEFAULT 'buyer', -- buyer | partner | admin
  referrer_id   bigint,               -- кто пригласил (ref_ deep link)
  lang          text DEFAULT 'ru',
  notify_daily  bool DEFAULT true,
  is_banned     bool DEFAULT false,
  created_at    timestamptz
)

partners (
  id            serial PK,
  user_id       bigint FK -> users,   -- владелец кабинета
  name          text,                 -- название бизнеса
  category      text,                 -- еда, красота, авто...
  address       text,
  discount_free    int DEFAULT 5,     -- % по скидке дня
  discount_premium int DEFAULT 10,    -- % для подписчиков (10–15)
  avg_check     int,                  -- средний чек, для счётчика «вы сэкономили»
  work_hours    text,
  is_active     bool DEFAULT true,    -- админ выключил
  is_paused     bool DEFAULT false    -- партнёр сам на паузе
)

subscriptions (
  id            serial PK,
  user_id       bigint FK -> users,
  status        text,                 -- pending | active | expired | rejected
  receipt_url   text,                 -- скрин чека в Storage
  amount        int DEFAULT 2000,
  paid_at       timestamptz,
  expires_at    timestamptz,          -- paid_at + 30 дней
  confirmed_by  bigint                -- admin id
)

daily_deals (
  id            serial PK,
  partner_id    int FK -> partners,
  deal_date     date UNIQUE,
  description   text
)

redemptions (                          -- активации скидок
  id            serial PK,
  code          text UNIQUE,           -- 6 символов, он же в QR
  user_id       bigint FK -> users,
  partner_id    int FK -> partners,
  type          text,                  -- daily | premium
  status        text,                  -- issued | used | expired
  issued_at     timestamptz,
  used_at       timestamptz,
  expires_at    timestamptz            -- issued_at + 30 мин
)

raffles (
  id            serial PK,
  title         text,
  prize         text,
  starts_at     timestamptz,
  ends_at       timestamptz,
  winner_id     bigint,
  status        text                   -- open | finished
)

raffle_entries (
  raffle_id     int FK,
  user_id       bigint FK,
  PRIMARY KEY (raffle_id, user_id)
)

broadcasts (
  id serial PK, text text, sent_at timestamptz, sent_count int
)
```

Индексы: `redemptions(code)`, `redemptions(partner_id, used_at)` — для статистики, `subscriptions(user_id, status)`.

## 3. Ключевые флоу

### 3.1 Оплата подписки (Kaspi, ручное подтверждение)
1. Пользователь жмёт «Оформить подписку 2000 ₸» → бот показывает номер Kaspi и просит скрин чека.
2. Пользователь шлёт фото → сохраняется в Storage, создаётся `subscription(status=pending)`.
3. Админу приходит карточка: чек + кнопки ✅ / ❌.
4. ✅ → `status=active`, `expires_at = now()+30d`, пользователю уведомление. ❌ → `rejected` + причина.
5. За 3 дня до `expires_at` — напоминание о продлении; после — `expired`, доступ закрывается.

Защита: не более 1 pending-заявки на пользователя; чек обязателен.

### 3.2 Активация скидки — механики

Четыре варианта, сравнение:

| Механика | Друг для кассира | Защита от фрода | Когда |
|---|---|---|---|
| **C. Наклейка QR на кассе, клиент сканирует сам** (deep link `t.me/bot?start=p_123`) | Ничего не делает | Средняя → высокая с защитами ниже. Наклейка = реклама бота + воронка регистрации | **MVP, основная** |
| **A. Клиент показывает код, кассир вводит** | Набрать 6 символов | Высокая: партнёр подтверждает визит вводом | MVP, фолбэк |
| **B. Кассир сканирует QR клиента** (Mini App `showScanQrPopup`) | Один тап + камера | Высокая | Фаза 2 |
| **D. Постоянная карта клиента** (личный QR, кассир сканирует → сервер сам даёт 5% или 10–15%) | Один тап | Высокая: кассир видит имя+фото | Фаза 2 |

Почему C основная: партнёры — узкое место модели. Наклейка не требует от кассира ничего, работает без смартфона у кассы, а каждый скан незарегистрированного прохожего = новый пользователь в воронке.

**Флоу MVP (вариант C):**
1. На кассе у партнёра наклейка с QR → клиент сканирует камерой телефона → открывается бот с `start=p_123`.
2. Не зарегистрирован → онбординг, затем продолжение. Зарегистрирован → бот проверяет подписку и лимиты.
3. Бот показывает **экран активации**: название заведения, % скидки, имя клиента, сегодняшняя дата + **живые тикающие часы** + «знак дня» (эмодзи/цвет, который бот утром рассылает партнёрам). Экран живёт 5 минут.
4. Клиент показывает экран кассиру → кассир сверяет знак дня и тикающие часы → даёт скидку.
5. Параллельно партнёру в бот прилетает пинг: «Иван активировал скидку 10%, 14:32» — кассир может свериться со своим телефоном (полностью исключает подделку экрана). Визит записан автоматически.

**Фолбэк (вариант A)** — для партнёров, которые хотят сами подтверждать визиты: клиент жмёт «Активировать» в боте, получает код `A7K2M9` (TTL 30 мин), кассир вводит его в кабинете и видит имя клиента. Одноразовость — атомарный UPDATE:
```sql
UPDATE redemptions SET status='used', used_at=now()
WHERE code=$1 AND partner_id=$2 AND status='issued' AND expires_at > now()
RETURNING *;
```

**Анти-фрод правила (общие):**
- Лимит: 1 активация у одного партнёра в день на пользователя (настраивается per-partner).
- Скидка дня: 1 активация на пользователя на конкретную скидку дня.
- Активация привязана к партнёру (deep link или код) — в другом заведении не работает.
- Экран активации: тикающие часы + дата + знак дня + имя владельца → скриншот и чужой экран палятся за секунды.
- Пинг партнёру в реальном времени — арбитр при любых сомнениях.
- Активации только в часы работы заведения (опционально, поле `work_hours`).

**Edge cases:**
- Отсканил дома «про запас» → экран живёт 5 минут, потом «Отсканируйте QR на кассе заново».
- Отсканил и ушёл без покупки → визит записан, статистика шумит. Партнёр видит пинги и может репортить; при желании партнёра — переключаем его на вариант A.
- Нет интернета у клиента на кассе → фолбэк A наоборот не поможет (тоже нужен интернет); кассир даёт скидку по подписке в профиле клиента (профиль кешируется ботом) или без скидки. Редкий случай.
- Несколько кассиров у партнёра → пинги приходят на один аккаунт партнёра; при необходимости — таблица `partner_staff`.
- Наклейку сфотографировали и повесили в другом месте → безвредно: активация всё равно записывается на партнёра из QR.

### 3.3 Кабинет партнёра
- Вход: админ добавляет партнёра по telegram_id → роль `partner`.
- Меню: «Ввести код клиента», «Статистика» (визиты за месяц, по дням), «Моя скидка» (изменить % в пределах 10–15).

### 3.4 Розыгрыш
- Админ создаёт розыгрыш; все пользователи получают кнопку «Участвовать».
- По `ends_at` планировщик выбирает случайного из `raffle_entries`, уведомляет всех.

### 3.5 Админка (в боте)
`/admin` → меню: партнёры (добавить/удалить), скидка дня (выбрать партнёра на дату), подписчики (список + поиск), заявки на оплату, розыгрыши, рассылка (с подтверждением перед отправкой; отправка батчами 25 msg/сек — лимит Telegram).

## 4.5 Фронтенд — Telegram Mini App (основной интерфейс)

**Решение (Фаза 2):** Mini App — основной интерфейс продукта. Бот — транспорт: онбординг (телефон + согласие), пинги партнёрам, уведомления, приём чеков Kaspi, Stars-платежи, фолбэк-активация.

**Стек:** React + Vite + TypeScript, UI — `@telegram-apps/telegram-ui` (нативный вид Telegram, авто-тема; требует React 18), SDK — `@telegram-apps/sdk-react`, карта — Leaflet + OpenStreetMap (без API-ключей; «Маршрут» — ссылка в 2GIS). Хостинг: фронт — Vercel, бэк (бот + API) — Railway. Роутинг — HashRouter (SPA на статике без настройки).

**API:** FastAPI в одном репо с ботом, общий слой `db.py` и `services/` (бизнес-логика активации — одна функция `redemption.activate()` для бота и API). Бот и API — два процесса, один docker-compose. Пинги/инвойсы из API — через aiogram `Bot` без polling.

**Авторизация:** Mini App шлёт `initData` в заголовке `Authorization: tma <initData>` → FastAPI валидирует HMAC-подпись (ключ = HMAC-SHA256("WebAppData", bot_token)) → telegram_id → роль из `users`. Паролей нет, подделка невозможна. Кэш валидации 5 мин, initData старше часа отклоняется.

| Функция | Бот | Mini App |

| Онбординг (телефон, согласие) | ✅ основной | → отправляет в бот |
| Активация по наклейке | фолбэк (`?start=p_id`) | ✅ основной (`?startapp=p_id`) |
| Каталог, карта, карточки | текстовый фолбэк | ✅ основной |
| Подписка: статус, история, «сэкономлено» | фолбэк | ✅ основной |
| Оплата Kaspi (чек) | ✅ (фото чека — только чат) | кнопка → в бот |
| Оплата Stars | обработка платежа | ✅ кнопка + openInvoice |
| Пинги партнёру, уведомления | ✅ только бот | — |
| Кабинет партнёра, ввод кода | фолбэк (`/code`) | ✅ основной |
| Админка: чеки, метрики, календарь | фолбэк (команды) | ✅ основной |
| Рассылки, /ban, добавление партнёров | ✅ только бот | Фаза 2б |

### Активация через Mini App (вариант C, основной флоу)

1. Наклейка на кассе: QR → `t.me/<bot>/app?startapp=p_123` (Direct Link Mini App).
2. Mini App открывается сразу → `POST /api/activate {partner_id}` → сервер применяет правила (подписчик — premium-% везде; без подписки — только у партнёра дня, иначе 402 → экран «оформите подписку») и пишет визит (`status=used`).
3. Экран активации: заведение, −%, имя, дата, знак дня, **живые тикающие часы по серверному времени** (offset от `server_time` — перевод часов на телефоне не поможет), гаснет через 5 минут.
4. Пинг партнёру в бот — как раньше.
5. Незарегистрированный: API вернёт 403 → экран «запустите бота» → онбординг в боте → бот-фолбэк активации доступен сразу из чата.

### Оплата Telegram Stars

`POST /api/stars-invoice` → `createInvoiceLink(currency=XTR)` → Mini App `invoice.open()` → бот ловит `pre_checkout_query` (отклоняет, если подписка уже активна) и `successful_payment` → подписка активна мгновенно, без админа. `subscriptions.payment_method='stars'`, `tg_charge_id` — для возвратов.
Trade-off: комиссия Telegram ~30% + вывод в валюте (нужна карта/крипта) vs Kaspi без комиссии, но с ручной проверкой. Цена в Stars — `SUBSCRIPTION_PRICE_STARS` (компенсирует комиссию).

## 5. Нагрузка и надёжность

Экибастуз ~120 тыс. населения → реалистично сотни–тысячи пользователей, десятки визитов/день. Это **ничтожная нагрузка**: один процесс бота (long polling, без webhook) + Supabase free tier хватит с запасом. Масштабирование не проектируем — незачем.

Надёжность:
- Docker + `restart: always` на VPS.
- Бэкап БД: Supabase делает сам (PITR на платном; на free — pg_dump кроном раз в день).
- Логи операций с деньгами (подтверждения подписок) — не удалять, только статусы.

## 6. Trade-offs

| Решение | Почему | Цена |
|---|---|---|
| Long polling вместо webhook | Не нужен домен/SSL, проще деплой | Ничего при такой нагрузке |
| Ручное подтверждение Kaspi | API переводов физлицам нет | Твоё время; при росте — Kaspi Pay API (нужно ИП + договор) |
| Наклейка QR на кассе как основная механика | Ноль действий от кассира, реклама бота, воронка регистрации | Скриншот-фрод (закрыт знаком дня + часами + пингом партнёру), шум в статистике |
| Всё в боте, Mini App потом | MVP за дни, а не недели | Витрина менее красивая на старте |
| Supabase, не голый Postgres | Storage для чеков + бэкапы + дашборд из коробки | Вендор; уйти легко (обычный Postgres) |

## 7. План MVP → полная версия

**Фаза 1 — MVP (1–2 недели):** регистрация, скидка дня, оплата чек→подтверждение, наклейки QR на кассах + экран активации + пинг партнёру, фолбэк с кодом, счётчик визитов, админ: добавить партнёра/скидку дня/список подписчиков. Дизайн и печать наклеек — в чеклист запуска.

**Фаза 2а (скелет готов):** Mini App — основной интерфейс: витрина + карта заведений, экран активации через `startapp`, профиль с оплатой Stars/Kaspi, кабинет партнёра, админка (чеки + метрики). API (FastAPI). Деплой: Railway (bot+api) + Vercel (Mini App).

**Фаза 2б:** геймификация (баллы/стрики), отзывы после визита, шеринг экономии, избранное+поиск, розыгрыши, календарь скидки дня в админке.

**Фаза 3 (при росте):** Kaspi Pay API (автоплатежи), stories партнёров, личный QR клиента, self-service партнёров, несколько городов (поле `city`).

**Пересмотреть при росте:** ручное подтверждение оплат (узкое место — ты), long polling → webhook при >10k пользователей, лимиты рассылки.
